# Product Proposal: AutoText2SQL Metadata Search

## 1) Обоснование идеи

Проект решает прикладную задачу: дать команде быстрый и безопасный поиск по структуре конкретного набора корпоративных БД (схемы, таблицы, колонки, связи), чтобы сократить время на ручной разбор и коммуникацию с инженерами.

Текущая боль:
- данные распределены по нескольким БД, и пользователям сложно понять, где именно хранится нужная сущность;
- названия таблиц и колонок не всегда очевидны для бизнес-терминов;
- ручной поиск по DDL, wiki и старым SQL занимает слишком много времени.

Идея:
- подключить фиксированный список БД;
- собрать и индексировать метаданные в едином knowledge-слое;
- дать интерфейс поиска по естественному языку с объяснением результата;
- выдавать простой read-only SQL-запрос для проверки гипотезы.

## 2) Цель и метрики успеха

### Цель
Создать агентную систему, которая находит релевантные таблицы/колонки/связи в фиксированном наборе БД быстрее и точнее текущего ручного процесса.

### Метрики успеха (с целевыми значениями)

| Тип метрики | Метрика | Как считаем | Целевое значение |
|---|---|---|---|
| Продуктовая | Доля успешных поисков | % запросов, где пользователь нашел нужную таблицу/колонку без эскалации | >= 75% |
| Продуктовая | Среднее время поиска | Среднее время от вопроса до полезного ответа | <= 20 сек |
| Агентская | Точность top-3 результатов | % запросов, где нужный объект БД есть в первых 3 результатах | >= 80% |
| Агентская | Доля правильных ответов на естественном языке | % запросов на естественном языке, где система выдала корректный и релевантный ответ по структуре БД | >= 70% |
| Техническая | p95 End-to-End Latency | 95-й перцентиль времени ответа системы | <= 8 сек |
| Техническая | Error Rate | % запросов с технической ошибкой (timeout/сбой сервиса) | <= 5% |

## 3) Потенциальные сценарии использования и edge-кейсы

### Базовые сценарии
1. Аналитик спрашивает: "В какой таблице хранится статус заказа и дата оплаты?"
2. Тестировщик ищет все таблицы, связанные с пользователем и его транзакциями.
3. Системный аналитик строит карту зависимостей между сущностями для ТЗ.

### Edge-кейсы
- Один термин соответствует разным сущностям в разных БД.
- Дублирующиеся названия таблиц в разных схемах.
- Частично устаревшие или неполные описания колонок.
- Сложные вопросы вида "как связаны A, B и C через промежуточные таблицы".

## 4) Ограничения (технические и операционные)

- Поддержка только заранее заданного набора БД (без автодобавления новых источников).
- p95 latency: до 30 секунд для типичного поискового запроса.
- SQL ограничен read-only и используется как вспомогательный запрос.


## 5) Архитектурный набросок

### Модули
1. Metadata Introspection
   - Сбор схем, таблиц, колонок, ключей, связей и описаний.
2. Search Agent
   - Интерпретация NL-запроса, retrieval релевантных объектов, ранжирование.
3. MCP/API Layer
   - Интерфейс интеграции с внешними инструментами.
4. Monitoring & Evals
   - Метрики качества поиска и наблюдаемость.

### Интеграции
- Реляционные БД c конкретного БД сервера.
- LLM Gateway для NL-понимания, маршрутизации по моделям и семантического сопоставления терминов.
- Векторное хранилище эмбеддингов метаданных БД и бизнес-терминов для семантического поиска.
- Система метрик и логов.

## 6) Потенциальный data flow

1. **Построение индекса**
   - Интроспекция БД -> нормализация метаданных -> индекс таблиц/колонок/связей.
2. **Обработка запроса**
   - NL-вопрос -> извлечение намерения и сущностей -> retrieval из индекса.
3. **Генерация пути поиска**
   - Формирование и выдача пути, по которому найдены данные: БД -> схема -> таблица.

### Что делегируется LLM/Agent, а что нет
- Делегируется:
  - разбор естественного языка;
  - сопоставление бизнес-терминов и объектов схемы;
  - формирование человеко-понятного объяснения результатов поиска.
- Не делегируется:
  - выполнение опасных SQL-операций;
  - принятие решений, нарушающих policy.
